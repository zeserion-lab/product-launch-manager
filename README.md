## 1. 概要

本アプリは、新商品の開発プロジェクトを**発売日を起点に逆算して管理する**業務支援ツールです。

社内で運用してきた Excel「新商品フロー」をベースに、発売日を入力するだけで企画〜発売までの全41工程の予定日程を自動計算します。発売日が変わっても手作業でスケジュールを修正する必要がなく、常に最新の日程で全工程を把握できます。

現時点では **個人利用・社内検証向けの試作版** です。業務への本格導入に向けたたたき台として活用することを想定しています。

---

## 2. アプリの目的

新商品の開発は、企画・試作・仕様・登録・製造・準備と多くの部署にまたがる複雑なプロジェクトです。このアプリは以下の課題を解決するために作りました。

| 課題 | このアプリでの対応 |
|---|---|
| 発売日変更のたびに Excel を手修正していた | 発売日を変えると全41工程が自動再計算される |
| どの工程が遅延しているか把握しにくかった | 予定日を過ぎた未完了タスクを自動で赤色ハイライト |
| 工程全体の進み具合が分かりにくかった | フェーズ別・全体の進捗率をリアルタイムで表示 |
| 複数案件を横断して状況を確認できなかった | ダッシュボードで全案件の状況を一覧比較 |

また、将来的な**複数案件管理・遅延アラート・チーム共有の土台**となるよう設計しています。

## 3. 主な機能

### ● スケジュール管理

- **発売日逆算による自動スケジュール**  
  発売日を入力するだけで、全41工程（企画〜発売まで233日分）の予定開始日・完了日を自動計算。発売日を変更すれば全工程の日程が即時再計算されます。

- **遅延タスクの自動検知**  
  予定完了日を過ぎた未完了タスクを自動で検出し、一覧上で赤色ハイライト表示。見落としを防ぎます。

### ● 進捗管理

- **ステータス管理**（未着手 / 進行中 / 完了 / 保留 / 遅延）
- **進捗率入力**（0〜100% のスライダー・数値入力）
- **担当者名・担当部署の記録**（テンプレートの担当部署を案件単位で変更可能）
- **メモ・実績日付の入力**（実績開始日・実績完了日）
- **フェーズ別進捗グラフ**（企画〜準備まで各フェーズの達成率をバー表示）
- **全体進捗率の表示**（ダッシュボード・案件詳細の両方で確認可能）

### ● タスク管理

- **タスク追加・削除**  
  標準41工程に加えて、案件ごとにカスタムタスクを追加できます。不要なテンプレートタスクは非表示にすることもできます。

- **タスク名称の変更**  
  テンプレートのタスク名を案件単位で上書きできます（元の名称はリセット可能）。

### ● 表示・検索

- **ガントチャート**  
  全工程をフェーズ別にタイムライン表示。今日の位置を赤ラインで示します。

- **フィルター・検索**  
  キーワード・フェーズ・担当部署・ステータス・遅延タスクのみ・重要工程のみで絞り込み。

- **ダッシュボード**  
  全案件のKPI（総案件数・進行中・発売間近・遅延あり・完了）と要注意タスク一覧を表示。

### ● データ管理

- **JSONバックアップ・インポート**（別PC・スマホへのデータ移行に対応）
- **ブラウザの localStorage に自動保存**（ネット接続不要）
- **スマートフォン対応**（ボトムナビ付きのレスポンシブデザイン）

## 6. データ保存と共有

### 保存の仕組み

データはユーザーの**ブラウザの localStorage** に保存されます。インターネット接続は不要で、外部サービスへのデータ送信もありません。

```
保存キー:
  plm:projects          → 案件の基本情報（名前・発売日・担当等）
  plm:tasks:{projectId} → 案件ごとのタスク実績（ステータス・進捗率等）
```

| 状況 | データの扱い |
|---|---|
| 同じブラウザで再アクセス | データは保持される |
| ブラウザの「閲覧データを削除」 | データが消える |
| 別のPC・スマホで開く | データは共有されない（手動移行が必要） |
| シークレットモードで使う | ブラウザを閉じるとデータが消える |

> ⚠ **大切なデータは定期的にバックアップを取ってください。**

---

### 別PC・スマホとデータを共有する方法

**サイドバー（またはスマホのボトムナビ）の「🔄 データ共有」ページから操作できます。**

```
① このPCで「📤 バックアップをダウンロード」
   → plm-backup-YYYYMMDD.json が保存される

② ファイルを送る
   メール / LINE / Googleドライブ / AirDrop など

③ もう一方の端末でアプリを開き「📥 インポート」
   → 送ったファイルを選択 → データが復元される
```

> ⚠ インポートすると移行先の既存データは上書きされます。  
> 必ず事前にエクスポートしてバックアップを取ってください。

---

### 将来の自動同期について

「保存したら全端末に自動反映」するには外部データベースが必要です。  
本アプリはそれを見越した設計になっており、**`src/storage/projectStorage.ts` の1ファイルを差し替えるだけで対応できます。**

```
現在（localStorage）         →  自動同期化後（例: Supabase）
────────────────────────────────────────────────────────
loadAll()                    →  GET /api/projects
saveProject(project)         →  PUT /api/projects/:id
saveTasks(id, tasks)         →  PATCH /api/projects/:id/tasks
deleteProject(id)            →  DELETE /api/projects/:id
```

## 7. 制限事項・現在の課題

本アプリは試作版であり、以下の制限があります。業務運用の前に確認してください。

| 制限事項 | 補足 |
|---|---|
| **試作版** | 実務運用を想定した検証版です。品質・安定性は随時改善予定 |
| **自動同期なし** | 端末間のリアルタイム同期は未対応。手動エクスポート/インポートで対応 |
| **ログイン・認証なし** | 誰でもアクセス・編集可能。社外公開には注意が必要 |
| **権限管理なし** | 担当者ごとの操作制限は未実装 |
| **遅延アラートなし** | 遅延検知は表示のみ。メール・Slack通知は未実装 |
| **DBなし** | データはブラウザに保存。サーバー側での保持・集計は不可 |
| **複数人同時編集不可** | 同一案件を複数名が同時に編集すると競合が発生する可能性あり |
| **スマートフォン最適化** | 基本的な表示には対応済みだが、複雑な操作はPCを推奨 |

## 8. 今後の拡張案

以下は今後の実装候補です。優先度の高い項目から順に記載しています。

| 優先度 | 機能 | 概要 |
|---|---|---|
| 🔴 高 | **自動同期（Supabase等）** | 保存と同時に全端末へ反映。`projectStorage.ts` 差し替えで対応可 |
| 🔴 高 | **遅延アラート・通知** | 期限N日前にSlack / メールで自動通知 |
| 🟡 中 | **担当部署別ビュー** | 「自分の担当タスクだけ表示」フィルター |
| 🟡 中 | **CSV / Excel 出力** | 週報・報告書向けのデータエクスポート |
| 🟡 中 | **コメント機能** | タスクへのスレッドコメント |
| 🟢 低 | **ユーザー認証・権限管理** | 担当者ごとのアクセス制御 |
| 🟢 低 | **履歴・変更ログ** | タスク更新の記録・差分確認 |
| 🟢 低 | **進捗更新の承認フロー** | 上長承認が必要な工程への対応 |
| 🟢 低 | **複数テンプレート対応** | 商品カテゴリ別の工程テンプレート |

## 10. ローカル起動方法

### 必要な環境

- **Node.js** 18 以上（[nodejs.org](https://nodejs.org) からインストール）
- **npm**（Node.js に同梱）
- Chrome / Edge / Firefox / Safari（最新版）

### 手順

```bash
# 1. リポジトリをクローン
git clone https://github.com/zeserion-lab/product-launch-manager.git
cd product-launch-manager

# 2. 依存パッケージをインストール
npm install

# 3. 開発サーバーを起動
npm run dev
# → ブラウザで http://localhost:5173/ を開く
```

### その他のコマンド

```bash
# 本番ビルド（Vercel デプロイ前の確認に使用）
npm run build

# ビルドしたファイルをローカルでプレビュー
npm run preview
# → http://localhost:4173/
```

## 12. 更新手順

コードを修正して公開に反映するまでの手順です。

```bash
# 1. Cursor でコードを修正

# 2. ビルドが通るか確認（エラーがないことを必ず確認）
npm run build

# 3. 変更内容をステージング
git add .

# 4. コミット（変更内容を簡潔に記録）
git commit -m "修正内容のメモ（例: feat: 発売日変更時のプレビュー改善）"

# 5. GitHub へ Push
git push

# → Vercel が自動検知して 1〜2分でデプロイ完了
# → 公開URL ( https://product-launch-manager.vercel.app/ ) にアクセスして確認
```

**確認ポイント**
- `npm run build` がエラーなく完了すること
- Vercel ダッシュボードの「Deployments」ステータスが `Ready` になること
- 公開URLで変更が反映されていること

## 13. 開発の背景・作成意図

本アプリは、社内で長年運用してきたExcel「新商品フロー」をもとに、より使いやすい形で業務に活かすことを目的として試作しました。

**開発のポイント**
- **Excelの良い部分（発売日逆算の考え方）を引き継ぎ、Webアプリとして再構築**
- **AIエディタ（Cursor）と ChatGPT を活用することで、エンジニアでなくても実装に参加**
- **「まず動くものを作る」を優先し、段階的に機能を追加していく方針で開発**
- **将来の本格運用（チーム共有・DB化・通知機能）を見越した設計を採用**

このアプリは現時点では試作版ですが、実際に業務で使いながら改善を重ねることで、**新商品開発の進捗管理ツールとして実務レベルに引き上げていく**ことを目指しています。
